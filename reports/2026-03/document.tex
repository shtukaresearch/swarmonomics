\section*{Current work}

\subsection*{Protocol strategy}
%
To coordinate future protocol updates, Swarm will need a framework to classify and triage initiatives by matching them to Swarm Network needs.
%
We propose the following ``pillars'':

\begin{itemize}
  \item \textbf{Aligning operator incentives.} 
    Calibrate incentives so as to ensure that the Swarm Network can store data safely and durably and serve retrievals with adequate performance for an origin storage use case, even if the Swarm Foundation does not operate any nodes.
  \item \textbf{Improve competitiveness of user-facing economic model.} 
    Identify realistic user/builder expectations about price level and predictability. Reduce cost drivers (while maintaining adequate performance and safety) and calibrate pricing mechanisms to a competitive range.
  \item \textbf{Reducing integration frictions.} 
    For protocol-level integration points such as onchain, improve DX by identifying and mitigating or eliminating frictions.
  \item \textbf{Measure and improve censorship resistance.} 
    Identify major failure modes of the storage service, whether unintentional or intentional and assess likelihood of each. 
  \item \textbf{Staker experience.} 
    Make it easier, safer, and more attractive to stake BZZ to earn mission-aligned ROI.

    (Enabling withdrawals and big nodes both contribute here.)
\end{itemize}

\subsection*{Withdrawals}

The scheduler design tracks stake balances using a breakdown into registered, onramping, offramping, and deregistered components (Figure~\ref{fig:balance-breakdown}).
%
Only registered balance contributes to revenue redistribution share.
%
Offramping balance must wait some time --- currently, we're thinking 28 days --- before tokens can actually be withdrawn.
%
During that time it is still productive, i.e.~continues to earn revenue share.
%
Deregistered tokens are unproductive: they're just sitting in the contract waiting to be withdrawn at any time (except while frozen, see below).

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.5\textwidth]{stake-balance-breakdown}
  \caption{Stake balance breakdown by registration status.}
  \label{fig:balance-breakdown}
\end{figure}

\

Several subtle design questions about how to handle interleaved wait and freeze periods were resolved.
%
The first set is about handling requests that arrive while the node is already waiting to exit.
%
Figure~\ref{fig:interleave-exit} illustrates two scenarios: scheduling an overlay change and scheduling a top-up while an exit is pending.
%
Allowing scheduled operations that would execute after an exit and end up being invalidated is clearly nonsensical.
%
Allowing operations that are scheduled and execute out-of-order before an exit is meaningful, but increases complexity and doesn't seem to enable any desirable behaviours.
%
For simplicity our design disallows both: once an exit is scheduled, no other operations can be interleaved.

Another is whether a node that has been frozen due to a consensus fault can schedule or execute a withdrawal.
%
If scheduling is allowed while frozen (Figure~\ref{fig:schedule-frozen-yes}), the withdrawal wait runs concurrently with the remaining freeze; 
%
otherwise the user must wait to add the operation to the schedule, then wait again to execute.
%
Allowing the waits to be concurrent does not invalidate the penalty effect of freezing or the filtering effect of a minimum wait period, so we allow interleaving in this case.

Similarly, if a withdrawal was already scheduled before the freeze, a question is whether the node can execute it while still frozen.
%
If execution is allowed (Figure~\ref{fig:execute-frozen-yes}), a node that is nearing the end of a wait to exit or withdraw faces nearly no penalty for the freeze. They may simply execute the withdrawal and redeposit under a new record, essentially evading the freeze.
%
Clearly, nodes should not be able to evade freeze penalties via ``exit scams,'' so this is \textbf{not} allowed.
